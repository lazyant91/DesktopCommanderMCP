# Protocol-Safe Stdio Transport

## Purpose

Local MCP Server uses stdio for all MCP JSON-RPC traffic. A single plain-text write to stdout can corrupt that stream and cause an MCP client to reject or disconnect the server.

`FilteredStdioServerTransport` keeps stdout protocol-safe by intercepting console output and non-JSON stdout strings. It converts them into MCP logging notifications instead of allowing arbitrary text to mix with protocol messages.

## Relevant files

- `src/custom-stdio.ts` — transport implementation
- `src/index.ts` — startup, configuration loading, handshake completion, and transport activation
- `src/server.ts` — MCP capabilities and request handlers
- `src/utils/logger.ts` — application logging facade

## Data flow

```text
Application code
    |
    | console.log / console.warn / console.error
    | non-JSON process.stdout.write
    v
FilteredStdioServerTransport
    |
    | notifications/message JSON-RPC
    v
stdout
    |
    v
MCP client
```

JSON-RPC strings generated by the MCP SDK pass through to the original stdout writer. Empty strings also pass through unchanged.

## Startup sequence

1. `src/index.ts` imports bootstrap code before filesystem work begins.
2. `FilteredStdioServerTransport` stores the original console methods and stdout writer.
3. Console methods and stdout writes are redirected.
4. Messages produced before MCP initialization are buffered with timestamps.
5. The server connects through the SDK transport.
6. `server.oninitialized` calls `enableNotifications()` after the client completes the handshake.
7. Buffered messages are replayed as `notifications/message` records in timestamp order.
8. Later console output is converted immediately.

This sequence prevents startup logs from appearing on stdout before the MCP client is ready to accept notifications.

## Console mapping

| Console method | MCP log level |
| --- | --- |
| `console.log` | `info` |
| `console.info` | `info` |
| `console.warn` | `warning` |
| `console.error` | `error` |
| `console.debug` | `debug` |

The logger name in notifications is `local-mcp-server`.

Example notification:

```json
{
  "jsonrpc": "2.0",
  "method": "notifications/message",
  "params": {
    "level": "info",
    "logger": "local-mcp-server",
    "data": "Configuration loaded successfully"
  }
}
```

## Plain stdout handling

When application code calls `process.stdout.write()` with a string:

- blank content passes through;
- content that appears to be JSON-RPC passes through;
- other text is routed as an `info` logging notification;
- non-string buffers pass through to the original writer.

The JSON-RPC check is intentionally small. It recognizes object strings containing common JSON-RPC fields such as `jsonrpc`, `method`, or `id`. Application code should not rely on this heuristic to construct protocol messages; normal MCP responses should be sent through the SDK.

## Public transport helpers

`FilteredStdioServerTransport` exposes:

- `enableNotifications()` — mark the handshake complete and flush buffered logs;
- `sendLog(level, message, data?)` — send or buffer a protocol-safe logging notification;
- `sendProgress(token, value, total?)` — send an MCP progress notification after initialization;
- `sendCustomNotification(method, params)` — send a custom JSON-RPC notification after initialization;
- `cleanup()` — restore original console methods and stdout writing;
- `isNotificationsEnabled` — report initialization state;
- `bufferedMessageCount` — report the number of startup messages waiting to be flushed.

These helpers are implementation facilities, not additional MCP tools.

## Serialization behavior

A single object argument is preserved as structured data. Multiple values are converted to strings and joined with spaces. Objects in multi-value logs are JSON-stringified when possible.

If serialization fails, the transport sends a small error notification rather than writing invalid text to stdout.

## Error channels

Normal application logging is converted to MCP logging notifications. Selected fatal fallback diagnostics may be written to stderr because stderr is outside the JSON-RPC stdout stream.

Clients differ in how they display logging notifications and stderr. Protocol safety does not guarantee that every client will render every message in the same way.

## Guarantees

The transport is designed to guarantee that:

- startup console output is deferred until initialization;
- ordinary console calls do not write raw text into the MCP stdout stream;
- logging notification payloads use the `local-mcp-server` identity;
- the original process-global functions can be restored by `cleanup()`.

## Non-guarantees

The transport is not:

- a persistent audit log;
- telemetry or analytics collection;
- a browser UI or progress dashboard;
- protection against native code or dependencies that bypass the patched JavaScript writer;
- a replacement for the MCP SDK's request and response transport.

## Testing

The test suite protects the current transport boundary by checking that:

- stdout filtering remains present;
- initialization enables notifications only after the handshake;
- removed client-specific dispatch and UI resources do not return;
- package builds contain only the current transport implementation.

Run:

```bash
npm test
npm run test:integration
```

Transport changes should also be tested with a real stdio MCP client on the target operating system.
